# external-secrets — ClusterSecretStore + IRSA wiring

The `borjie-aws` ClusterSecretStore is the single source of truth for
every runtime secret consumed by BORJIE workloads (every per-service
ExternalSecret references it). This directory holds the cluster-scope
manifests required to bootstrap that store.

## Layout

| File                                | Purpose                                        |
|-------------------------------------|------------------------------------------------|
| `service-account.yaml`              | IRSA-annotated SA for the ESO controller pod   |
| `cluster-secret-store-aws.yaml`     | The `borjie-aws` ClusterSecretStore        |
| `ghcr-pull-externalsecret.yaml`     | Fanned-out `ghcr-pull` Secret for image pulls  |
| `kustomization.yaml`                | Kustomize entry point                           |

## Prerequisites

1. **External Secrets Operator** installed in the `external-secrets`
   namespace via the official Helm chart:
   ```bash
   helm repo add external-secrets https://charts.external-secrets.io
   helm install external-secrets external-secrets/external-secrets \
     -n external-secrets --create-namespace
   ```
2. **EKS cluster with IRSA enabled** (OIDC identity provider associated):
   ```bash
   eksctl utils associate-iam-oidc-provider --cluster <cluster> --approve
   ```
3. **IAM role + policies** as described below.

## Apply

```bash
# 1. Bootstrap (once per cluster)
kubectl apply -k infra/k8s/external-secrets
# 2. Then any application overlay
kubectl apply -k infra/k8s/overlays/staging
```

## AWS region

The store points at `af-south-1` (Cape Town). For BORJIE, this is the
TZ-closest AWS region with full Secrets Manager + EKS support. If your
cluster lives in another region, override at the Helm/operator layer
(per-cluster) rather than per-environment — staging and prod within a
single cluster should never split regions for the secret backend.

## IAM trust policy (for the role `eks-external-secrets`)

The role assumed by the `external-secrets` SA via IRSA. Replace
`ACCOUNT_ID`, `REGION`, and `OIDC_PROVIDER_ID` (without protocol prefix):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::ACCOUNT_ID:oidc-provider/oidc.eks.REGION.amazonaws.com/id/OIDC_PROVIDER_ID"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "oidc.eks.REGION.amazonaws.com/id/OIDC_PROVIDER_ID:sub": "system:serviceaccount:external-secrets:external-secrets",
          "oidc.eks.REGION.amazonaws.com/id/OIDC_PROVIDER_ID:aud": "sts.amazonaws.com"
        }
      }
    }
  ]
}
```

## IAM permission policy (attached to the role)

Scoped to BORJIE's secret tree only. Replace `ACCOUNT_ID`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ReadBorjieSecrets",
      "Effect": "Allow",
      "Action": [
        "secretsmanager:GetSecretValue",
        "secretsmanager:DescribeSecret"
      ],
      "Resource": "arn:aws:secretsmanager:af-south-1:ACCOUNT_ID:secret:borjie/*"
    },
    {
      "Sid": "ListBorjieSecrets",
      "Effect": "Allow",
      "Action": "secretsmanager:ListSecrets",
      "Resource": "*"
    }
  ]
}
```

## Image-pull secret (`ghcr-pull`)

Until the GHCR packages are public, every Deployment / CronJob carries
`imagePullSecrets: [{ name: ghcr-pull }]`. The `ghcr-pull` Secret is
generated by ESO in each application namespace from
`borjie/shared/ghcr-pull-token` (Docker config JSON).

Seed the AWS secret once:

```bash
PAT="ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"  # `read:packages` scope
USER="GeorgeMwiki"
AUTH=$(printf '%s:%s' "$USER" "$PAT" | base64)
JSON=$(printf '{"auths":{"ghcr.io":{"auth":"%s"}}}' "$AUTH")
aws secretsmanager create-secret \
  --name borjie/shared/ghcr-pull-token \
  --region af-south-1 \
  --secret-string "{\"dockerconfigjson\":\"$(echo -n "$JSON" | base64)\"}"
```

If you later flip the packages to public visibility in GitHub
(Packages → Settings → Change visibility), the `imagePullSecrets` stanza
becomes a no-op and you can drop both this ESO file and the per-manifest
references in a follow-up PR.

## Items still pending human input

- AWS account ID (placeholder `ACCOUNT_ID` in
  `service-account.yaml` and in the IAM policy templates above)
- IAM role ARN (`arn:aws:iam::ACCOUNT_ID:role/eks-external-secrets`) —
  create the role with the trust + permission policies above, then patch
  the SA annotation
- OIDC provider ID (`OIDC_PROVIDER_ID`) — visible in the EKS cluster
  details, format is a 32-char hex string
- The `borjie/shared/ghcr-pull-token` secret must be seeded in AWS
  Secrets Manager before any namespace overlay is applied
