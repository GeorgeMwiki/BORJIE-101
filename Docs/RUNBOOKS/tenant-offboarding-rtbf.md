# Tenant Offboarding & Right-To-Be-Forgotten (RTBF) Runbook

> Audience: tenant-ops, DPO, on-call platform engineer.
> Regulatory anchors: GDPR Art. 17 ("Right to erasure"),
> TZ PDPA 2022 Art. 28 (data-subject rights),
> CCPA §1798.105 (right to delete).

## Scope

This runbook covers two related flows:

1. **Tenant offboarding** — full tenant teardown (account closure,
   contract end, fail-to-pay enforcement).
2. **RTBF / DSAR-erase** — single data-subject erasure request,
   tenant continues operating.

Both flows traverse the same downstream cascade but differ in scope
and consent capture.

## Step 0 — Consent capture

Every erasure starts with a recorded request. No verbal-only erasures.

### Self-serve (preferred for RTBF)

The customer portal exposes `POST /api/v1/gdpr/erase-request`:

```bash
curl -X POST "$API_BASE_URL/api/v1/gdpr/erase-request" \
  -H "Authorization: Bearer $CUSTOMER_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "scope": "self",
    "reason": "Withdrawing consent under GDPR Art. 7(3)",
    "confirmationToken": "<emailed-verification-token>"
  }'
```

The endpoint:
- Re-verifies identity via a one-time emailed token.
- Writes a `gdpr_erase_request` row (status `pending`, 30-day grace).
- Sends a confirmation email with a recall link.

### Operator-initiated (offboarding)

For tenant-wide offboarding, the platform admin uses the HQ tool
`platform.offboard_tenant` which is `tier=destroy` + four-eye-approval.
The plan-artifact MUST include:

- Tenant id, legal name, registered contact.
- Reason (contract end / fail-to-pay / RTBF cascade).
- Effective date.
- Backup-export decision (operator OR customer chooses).

## Step 1 — 30-day grace period

Every erasure has a hard 30-day pause window. During this period:

- The tenant is flagged `pending_erase` — login still works, writes
  are read-only banner-gated.
- Daily reminder emails sent to the registered contact.
- A recall endpoint is exposed:
  ```
  POST /api/v1/gdpr/erase-request/:id/recall
  ```
- Auto-resume on recall; no further action needed.

At T+30 days the cascade fires automatically via the
`gdprEraseCron` supervisor.

## Step 2 — Pre-erasure backup export

The customer is offered an exhaustive export of their data BEFORE
deletion. The export bundle covers:

- Personal data tables (PII columns decrypted client-side via the
  customer's session key)
- Payment receipts (PDF)
- Lease + inspection documents
- Conversation transcripts (kernel turns scoped to the user)
- Audit-trail entries the customer is the subject of

Generated by the DSAR compiler (`packages/database/src/services/dsar.service.ts`).
Default format: signed ZIP delivered via signed URL (24h TTL).

## Step 3 — Downstream cascade (the 13 + extras)

The DSAR compiler cascades erasure across every table that touches
the subject:

| # | Table | Strategy |
|---|---|---|
| 1 | `users` | Hard-delete |
| 2 | `tenant_users` | Hard-delete |
| 3 | `pii.email_enc`, `pii.phone_enc`, `pii.id_number_enc` | Hard-delete |
| 4 | `customer_profiles` | Hard-delete |
| 5 | `leases` (where tenant = subject) | Anonymize tenant_id → `erased-<uuid>` |
| 6 | `payments` | Anonymize payer_name + decrypted-detail columns |
| 7 | `notifications_log` | Hard-delete user_id-scoped rows |
| 8 | `kernel_memory_episodic`, `_semantic`, `_procedural`, `_reflective` | Hard-delete rows where subject is in payload (FK + JSON scan) |
| 9 | `kernel_action_audit` | Anonymize actor_id (audit chain preserved; row hash unchanged because canonical form replaces subject id with sentinel) |
| 10 | `inspections` (subject as inspector or tenant) | Anonymize |
| 11 | `documents_intake` (uploaded by subject) | Hard-delete metadata + S3 object |
| 12 | `voice_turns`, `voice_sessions` | Hard-delete |
| 13 | `feedback_signals` | Hard-delete |

Plus the post-Phase-B additions:

- **S5/B5 session-replay chunks** — every replay chunk
  (`packages/observability/session-replay/*` + S3 bucket) is purged.
- **CoT-reservoir samples** — `kernel_cot_reservoir` rows pinning
  the subject's reasoning are purged.
- **Reflexion buffer** — `kernel_reflexion_buffer` rows scoped to
  the subject are purged.

The cascade runs in a single DB transaction per table; cross-table
ordering follows FK dependency. Failures roll back per-table and
write a `gdpr_erase_failure` row for retry.

## Step 4 — Backup purge

Erased data must be purged from backups too. Cross-link:
`Docs/RUNBOOKS/backup-restore.md` — § "RTBF-driven backup pruning".

Strategy options (operator chooses at offboarding time):

| Strategy | Recovery window | RTBF compliance |
|---|---|---|
| Synthetic-substitution in backups | Full | Strongest — subject row absent from any restore |
| Tombstone-then-restore-rewrite | Full | Strong — restore replays erasure on bring-up |
| Aged-out (wait for 90-day backup retention to expire) | Partial | Weakest — only legal when contract allows |

Most tenants ship synthetic-substitution. Document the choice in the
DSAR record.

## Step 5 — Evidence export (auditor-facing)

Generate an erasure proof bundle:

```bash
pnpm -C scripts ts-node gdpr-evidence-export.ts \
  --request-id $REQ_ID \
  --out /tmp/erasure-evidence-$REQ_ID.zip
```

Bundle contents:

- Original consent capture (request payload, email confirmation log).
- Per-table cascade summary (counts erased, counts anonymized).
- `kernel_action_audit` rows showing the cascade run (with row hashes).
- Backup-purge attestation signed by the storage backend.
- Final hash-chain verification proving no audit-tamper occurred during cascade.

## Special cases

### Subject is a property owner with active tenants

Owner offboarding does NOT cascade to tenant data. Tenant data is
the tenant's own subject scope. Owner-scoped data (payment routing,
KYC) is erased per the standard cascade.

### Pending payments at erase time

Payments in flight are settled first. The cascade waits for
`payments.status IN ('settled','failed')` on every row. If a
payment is stuck `pending` > 24h, the cascade flags
`gdpr_erase_blocked` and surfaces to the DPO for manual triage.

### Court-order data hold

Erasure is paused if `legal_hold` is set on the tenant. DPO clears
the hold before cascade resumes. Document the hold reference.
