# Geo intelligence — SOTA 2026-05-29

**Audience:** Borjie engineering + product.
**Scope:** Drive every geo-touching surface in Borjie to production-grade
state-of-the-art. Mining lives on geography: sites, hazard zones,
transport routes, regulatory regions, neighbouring estates,
chain-of-custody waypoints. The geo stack is load-bearing for safety,
royalty truth, evidence integrity, and marketplace liquidity.

This is the synthesis grounding the 0130 migration, the geofencing
service, the brain-tool catalog, and the `@borjie/maps` package.

## §1 — The spatial database: PostGIS 3.x, no exceptions

### Why PostGIS

PostGIS turns Postgres into a full OGC-compliant spatial DB. For the
Borjie workload (sites, polygons, points, distance queries, point-in-
polygon, route distance) it dominates every alternative we considered:

- **Native types**: `geography(POINT,4326)`, `geography(POLYGON,4326)`,
  `geometry(MULTIPOLYGON,4326)` with proper EPSG handling.
- **Spatial indexes**: `GiST` indexes on geometry columns deliver
  sub-millisecond point-in-polygon and distance queries up to 10M rows.
  `SP-GiST` is preferred for point clouds; `BRIN` for time-series GPS
  trails (workforce_locations).
- **Query patterns**: `ST_DWithin`, `ST_Contains`, `ST_Intersects`,
  `ST_Distance`, `ST_Buffer`, `ST_MakePoint`, `ST_GeographyFromText`.
- **JSONB compatibility**: `ST_AsGeoJSON` round-trips with our existing
  `text` columns at the Drizzle ORM boundary so we do not have to
  rewrite every handler — schemas keep a `text` (GeoJSON) face while
  the migration adds a real geometry shadow column + GiST index.

### Why not the JSONB-only path

We already store GeoJSON in `geo_nodes.polygon` and `sites.polygon`
columns. Doing point-in-polygon in app code at scale is unworkable:

- A single 1k-vertex polygon scan in JS takes ~3ms; doing it against
  500 hazard zones for every clock-in tick blows our latency budget.
- We cannot index JSON polygons. Every `is-this-worker-in-a-hazard`
  read becomes a full-table scan.
- ETL becomes a nightmare when regulator tooling expects GeoPackage or
  shapefile imports.

### Migration shape (idempotent, additive)

```sql
CREATE EXTENSION IF NOT EXISTS postgis;

-- Sites: add geometry shadow columns mirroring text/GeoJSON columns
ALTER TABLE sites
  ADD COLUMN IF NOT EXISTS location_geom geography(POINT, 4326),
  ADD COLUMN IF NOT EXISTS polygon_geom  geography(POLYGON, 4326);

CREATE INDEX IF NOT EXISTS sites_location_gix
  ON sites USING GIST (location_geom);
CREATE INDEX IF NOT EXISTS sites_polygon_gix
  ON sites USING GIST (polygon_geom);
```

Drizzle stays oblivious to the new columns; raw SQL in the geofencing
service uses them. When the text/GeoJSON column updates we also update
the geometry shadow via a Postgres trigger or via the same handler.

## §2 — Geofencing: the canonical Borjie pattern

### Core domain operations

The geofencing service exposes pure spatial predicates plus stateful
event emission:

1. **pointInSite(lat, lon, tenantId) -> SiteHit | null**
   Return the site whose polygon contains the point. Tenant-scoped.
2. **pointInHazard(lat, lon, tenantId) -> HazardHit[]**
   Return all hazard zones containing the point ordered by severity.
3. **pointInTitle(lat, lon, tenantId) -> LicenceHit | null**
   Return the mining title polygon the point falls inside. Used for
   trespass detection and royalty calc.
4. **distanceToNearestSite(lat, lon, tenantId) -> Distance**
   `ST_Distance(loc::geography, sites.location_geom)` ordered.
5. **pointInComplianceZone(lat, lon) -> RegulatoryZone[]**
   PCCB region + NEMC catchment + EITI boundary lookup. Tenant-agnostic
   because regulators publish the same boundaries to everyone — we
   store them with `tenant_id = NULL` exactly like the intelligence
   corpus.

### Watcher worker — `geofence-watcher`

- Ticks every 30s; reads from `workforce_locations` (new ephemeral
  table) whose rows are TTL'd 24h.
- For every active employee with a recent fix, runs:
  - `pointInSite` — are they at the right site for their shift?
  - `pointInHazard` — are they inside a `forbidden` polygon?
  - If outside the assigned site for >5 min → emit `worker_offsite_alert`.
  - If inside a hazard polygon → emit `worker_in_hazard_alert`
    (audit hash chained; HIGH severity for forbidden zones).

The watcher follows the established `cases-sla-supervisor.ts` shape:
multi-tenant loop, sequential ticks, `Logger` only (Pino), gated by
DATABASE_URL + env flag, idempotent start/stop.

### Reference geofencing patterns

PostGIS geofencing is the same pattern used by Uber, Lyft, DoorDash,
Strava, and every fleet-tracking company since 2012. The 2025 best-
practice synthesis:

- Pre-filter with `&&` bounding box before `ST_Contains` for
  10–100× speedup on big polygons.
- Use `geography(...,4326)` for distance work (metres) and
  `geometry(...,4326)` for shape work (degrees) — convert at the edge.
- Never store lat/lon as separate numerics if you also have a geometry
  column. The numerics drift and become wrong.

## §3 — Map UI: MapLibre GL is the right default

### Why MapLibre, not Mapbox or Leaflet

| Concern              | MapLibre GL | Mapbox GL | Leaflet  |
|----------------------|-------------|-----------|----------|
| License              | BSD-2       | Proprietary v2+ | BSD-2 |
| Vector tile / WebGL  | Yes         | Yes       | No (raster) |
| Performance @ 5k pts | 60 fps      | 60 fps    | 5 fps    |
| Mobile parity (RN)   | MapLibre RN | Mapbox RN (paid) | n/a   |
| Cost                 | $0          | $0.50/1k tile + per-MAU | $0  |
| 2026 momentum        | 50+ corporate sponsors | Stable | Niche |

Mapbox switched to a non-OSS license in late 2020 (v2). MapLibre GL is
the open-source fork, evolving in parallel with broader corporate
backing. For Borjie — a Tanzania-first mining product where every
MAU-fee charged in USD is a tax on our margin — MapLibre is the only
defensible choice.

Leaflet stays as a fallback for very low-spec Android only; the
WebGL gap to MapLibre is too large to use Leaflet on a B2B map with
hundreds of features.

### Package architecture: `@borjie/maps`

Single workspace package, two entry points:

- `@borjie/maps/react` — `<BorjieMap>`, `<BorjieSiteLayer>`,
  `<BorjieHazardLayer>`, `<BorjieClusterLayer>`. Built on `react-map-gl`
  (Uber/Vis.gl) which is the mature React wrapper for both MapLibre
  and Mapbox.
- `@borjie/maps/native` — exports the same component names backed by
  `@maplibre/maplibre-react-native`. Component contract stays identical
  so the Expo screens reuse the React typings without code splits.

Both entry points are token-free by default (uses
`https://demotiles.maplibre.org/style.json`), with an
`EXPO_PUBLIC_BORJIE_MAP_STYLE_URL` env var for self-hosted MapTiler
or Stadia tiles. Bilingual sw/en place labels come from MapTiler's
"name:sw" / "name:en" tile fields (free tier covers TZ).

Dark mode comes via `style.url` swap; both light and dark Tanzania-
focused styles ship from-day-one. Polygon styling and marker icons
follow the existing Borjie design tokens.

## §4 — Chain-of-custody geo: C2PA + EXIF + signed GPS

### The standard

C2PA (Coalition for Content Provenance and Authenticity, 6,000+
members in 2026) wraps every artefact in a tamper-evident manifest
that includes SHA-256 + X.509 signing of every assertion — including
GPS, EXIF, and edit-history. Hardware-signed capture (Leica SL3-S
Nikon Z6 III, Canon EOS R5 Mark II from January 2025) gives the
strongest provenance: the camera itself signs at capture time so any
in-flight tamper attempt breaks the chain.

For Borjie, we don't ship hardware-signed cameras to artisanal miners.
Our pragmatic chain is:

1. Mobile app captures photo + GPS + accuracy + heading at the same
   instant via `expo-location` + `expo-camera`.
2. App signs an in-memory manifest:
   `{ photo_sha256, gps:{lat,lon,acc,heading}, captured_at, device_id }`
   with the device's per-install signing key (the same one we mint
   for biometric clock-in).
3. Upload pushes photo + manifest as one indivisible blob into the
   evidence store.
4. Server stores the manifest into the chain-of-custody row, hash-
   chains into `mineral_chain_of_custody.prev_audit_hash`.
5. Every CoC step gets a `geo_at_step` jsonb column carrying
   `{lat, lon, accuracy_meters, heading_deg, captured_at}` straight
   from the manifest.

### Buyer-facing trace

The buyer-mobile UI shows the parcel's CoC as a polyline on a map: pit
location → transit fix(es) → assayer site → smelter location → buyer
delivery. Every node is clickable and surfaces the signed manifest +
who attested it. This is the marketplace differentiator: every other
mineral marketplace has paperwork; we have a cryptographically signed
spatial timeline.

## §5 — Tanzania regulatory geo

The three boundary sets we need:

1. **PCCB regions** — Prevention and Combating of Corruption Bureau
   regions. We use these for trespass alerts on government-restricted
   zones and for the four-eyes approval routing of high-stakes brain
   actions.
2. **NEMC catchment areas** — National Environment Management Council
   river / watershed / forest reserve catchments. Mining inside a
   protected catchment area is a §29 EMA offense; the geo zone of a
   site determines which NEMC officer routes the EIA.
3. **EITI reporting boundaries** — TEITI 14th report (2024) groups
   small-scale mining into 10 zones. Royalty + local-content
   reporting follows these zones; the EITI website publishes the
   shapefile annually.

The 10 small-scale mining zones (TEITI):

- Arusha-Manyara
- Bukoba
- Dodoma
- Mbeya
- Mpanda
- Mtwara
- Musoma-Mwanza
- Shinyanga-Songwe
- Singida
- Tanga-Kilimanjaro

We seed these as polygon stubs in migration 0130 (covering the named
admin regions per Tanzania NBS 2022 boundaries) and refine with the
official TEITI shapefile when it becomes machine-readable.

### Endpoint surface

`GET /v1/regulatory/zones/by-point?lat=X&lon=Y`
→ Returns the PCCB region, NEMC catchment, and EITI zone that contain
the point. Tenant-agnostic. Cached 24h. Used by every brain tool that
needs to ask "which jurisdiction does this fall under?".

## §6 — Brain tools we ship

Five geo tools added to the brain catalog, persona-gated:

| Tool ID                                | Persona  | Purpose |
|----------------------------------------|----------|---------|
| `mining.geo.site.nearby`               | owner/manager | Owner's sites within R km of a point. |
| `mining.geo.title.contains`            | owner/admin   | Is point inside any mining title polygon? |
| `mining.geo.hazard.proximity`          | manager/worker| Risk score from nearest hazard distance. |
| `mining.geo.compliance.zone_of`        | owner/admin   | PCCB + NEMC + EITI zone for point. |
| `mining.geo.route.optimize`            | manager       | Tanzania-aware route from A→B avoiding impassable roads in wet season. |

Each tool follows the existing `PersonaToolDescriptor` shape, calls
the internal loopback HTTP client to hit `/v1/...` endpoints owned
by the geofencing service, and emits audit-chain entries on every
write. None are writes — all five are READ tools (LOW stakes).

## §7 — What we keep verbatim from existing code

- `useLocation.ts` (workforce-mobile) — the foreground-permission
  capture pattern is already correct; we keep the haversine helper
  in `fence.ts` as the fast-path for offline mobile checks.
- `clock-in-events.schema.ts` — the geo_lat / geo_lng numeric columns
  are kept; we add a geometry shadow on the table for clock-in fraud
  trigger queries.
- `mineral_chain_of_custody` — has the right hash-chain shape; we add
  a `geo_at_step jsonb` column for the signed GPS manifest.
- The existing `geo.schema.ts` and `geo-routing.schema.ts` JSONB
  polygons stay where they are; they're per-org route hierarchies for
  the customer-service workflow and not the mining-domain spatial
  store. We do not unify the two — distinct domain models, distinct
  storage.

## §8 — Out-of-scope (deliberate)

- **3D photogrammetry tiles** (Google 3D Tiles / Cesium) — owner
  cockpit "fly to land" demo is a future Wave; not load-bearing
  today.
- **Surveyor-grade COGO from cadastral docs** — Esri ArcGIS Parcel
  Fabric territory. We integrate with e-Ardhi when the API ships;
  we do not build our own parcel fabric.
- **Offline map tile bundling** — useful for remote sites with no
  connectivity, but adds 200MB+ to the mobile bundle. Phase 2.
- **Routing engine deeper than `route.optimize` stub** — true
  Tanzania-aware routing requires a per-season-edge-weighted OSRM
  build. Stub returns straight-line distance + a wet-season penalty
  factor; we polish in Phase 2.
