# PIECE L — Brain ↔ Tab Loop Architecture

**Status:** Decision-grade draft for Wave 22+. Foundation doc for all later loop-wave agents.
**Owner-vision quote:** *"User can go to tabs and use as normal OS, just still super-powered by intelligence... OR just chat with brain forever, but find all data captured and organized accordingly in the respective tabs."*

---

## Section 1 — The Loop in 6 hops

### Hop 1 — User chats with brain
Brain entry stays `kernel.think(req)` at `packages/central-intelligence/src/kernel/kernel.ts` (14-step pipeline). Persona-tier comes from `selectPersona` in `kernel/identity.ts`. Tier examples:
- **T1 owner** ("How is Mbezi performing vs Kariakoo this quarter?") — strategic; activates head-briefing agents, LATS for multi-step reasoning.
- **T3 estate-manager** ("Mr. Juma's unit 4B leaks again, schedule plumber Thursday") — operational; intent = `propose_action` with a concrete asset reference.
- **T5 customer** ("When does my lease expire?") — informational; intent = `request_info`; resolved by deterministic retrieval, no proposal generated.

Hop 1 is unchanged code — kernel + agent-loop already work today.

### Hop 2 — Conversational capture
After kernel `think()` finalises, a new post-pipeline stage `capture()` runs over the just-completed `(user_msg, assistant_msg)` pair. It produces a `conversation_capture` row per qualifying turn. Reuses three existing primitives:
- `packages/ai-copilot/src/conversation-state/entity-extractor.ts` (regex NER for property_id, unit_id, amount, dates, districts — already exists, zero LLM cost).
- `packages/ai-copilot/src/progressive-intelligence/extraction-patterns.ts` + `field-mappings.ts` (LLM-augmented field extraction).
- `packages/ai-copilot/src/conversation-state/state-machine.ts` (intent inference).

New work: an `IntentClassifier` (Haiku, low-cost) emits one of `{request_info, propose_action, file_event, ask_for_help, ambiguous}`. When intent ∈ `{propose_action, file_event}`, a candidate `ActionPlan` is generated by reusing `packages/central-intelligence/src/kernel/agency/goals/plan-decomposer.ts`.

All four outputs (entities, intent, action-candidates, confidence) are stored in a single immutable `conversation_capture` row keyed by `message_id` from Piece F's `conversation_messages`.

### Hop 3 — Dispatcher routes captures to tabs/modules
A new pure function `dispatchToTabs({tenantId, capture, sessionContext})` lives at `packages/central-intelligence/src/kernel/dispatch/router.ts`. It:
1. Looks up `routing_rules` (entity_type × intent → module_template_id + action).
2. Resolves entities to canonical `core_entity` rows via the Piece K resolver (`packages/ai-copilot/src/document-analysis/document-intelligence.ts` exposes the resolver port already).
3. Generates one `module_update_proposals` row per matched rule.
4. Computes confidence = `min(captureConf, resolverConf, ruleConf) × tenantTrustFactor`.
5. Below `policy_gate.autoWriteThreshold` (default 0.78) → proposal stays `status='pending_hitl'` and emits a notification through `packages/observability/src/event-bus.ts`. Above threshold AND inviolable-clean → status='auto_applying', the executor at `packages/central-intelligence/src/kernel/agency/executor/executor.ts` is asked to fire the module's accept-proposal handler.

Module-side acceptance: each module template (ESTATE, HR, FLEET, etc.) exposes a Zod-validated handler `accept_proposal(proposal_id) → AcceptResult` registered through `packages/agent-platform/src/agent-card.ts`. The handler is the ONLY path that mutates module tables from a capture — keeps audit clean.

### Hop 4 — Tabs reflect new data
Tabs subscribe to accepted-proposal streams via Supabase realtime channels keyed `tenant:{id}:module:{tplId}:proposals`. Subscription registry is the new `tab_subscriptions` table (static-ish; mostly a per-tenant-per-module enable flag).

The adaptive layout engine at `packages/dynamic-sections/src/lib/adaptive-layout/engine.ts` gets a new `proposal-policy` layered alongside the existing four policies (`frustration-policy`, `intent-policy`, `recency-policy`, `role-mastery-policy`). A pending proposal boosts the affected section; an accepted proposal triggers a one-frame "highlight" treatment in `components/SectionMount.tsx`.

Optimistic UI: `module_update_proposals` rows with `status='pending_hitl'` render as a translucent overlay row in the relevant section ("Lease renewal proposed by Brain — Approve / Decline / Edit"). Live cursor: real-time `postgres_changes` channel pushes new rows; React Query keys at `packages/dynamic-sections/src/hooks/query-keys.ts` are invalidated.

### Hop 5 — In-tab AI assist (the OS feel)
Every tab opens with the same kernel attached as a side-rail copilot, but the tool catalogue is module-scoped at `createToolRegistry`. The side-rail receives a `module_context: {moduleTemplateId, viewedRecordIds[]}` slice that the kernel injects into `awareness-scopes.ts` so retrieval is bounded to that module's slice of `core_entity`.

User actions inside the tab (click, edit, filter, export) emit OTel spans tagged `borjie.tab_event=true` AND append to a new `tab_event_log` table. The kernel's working memory reads `tab_event_log` on the next think() to keep continuity. Selecting a record exposes a "Why is this here?" button → traverses `conversation_capture.message_id` and (when present) `documents.id` to render an origin card. Bulk operations call into the brain for synthesis (e.g. "summarise these 12 maintenance tickets") — same kernel call, same audit chain.

### Hop 6 — Tab events feed back into brain memory
Two paths run continuously:
1. **Hot path:** every `tab_event_log` insert publishes to `event-bus.ts`; a small subscriber in `packages/ai-copilot/src/ambient-brain/behavior-observer.ts` updates the in-memory "behaviour profile" so the brain's next reply can reference the user's tab activity ("Saw you opened 3 arrears cases — want a triage list?").
2. **Cold path:** the existing sleep-consolidation pipeline at `services/consolidation-worker/src/stages/` already does ingest → cluster → reflect → promote → decay → consolidate → re-embed → publish. We add one stage `04c-tab-events-fold.ts` that folds `tab_event_log` into `semantic_memory`. No new infrastructure — same orchestrator, same OTel tracer.

## Section 2 — Data flow + schema

Four new tables, all under the gold-standard RLS pattern from `packages/database/src/migrations/0185_decision_traces.sql` (ENABLE + FORCE RLS, `tenant_isolation_select` USING `current_app_tenant_id()`, `tenant_isolation_modify` FOR ALL, REVOKE ALL FROM anon). Suggested migration numbers: 0215-0218.

### `conversation_capture` (mig 0215)
```
id              TEXT PK
tenant_id       TEXT FK tenants(id) NOT NULL
message_id      TEXT FK conversation_messages(id) UNIQUE NOT NULL   -- Piece F
intent          TEXT NOT NULL  -- enum-as-text
entities        JSONB NOT NULL DEFAULT '[]'  -- array of {type,value,canonical_entity_id,confidence}
action_candidates JSONB NOT NULL DEFAULT '[]'
capture_confidence NUMERIC(3,2) NOT NULL
extractor_version TEXT NOT NULL  -- prompt version pinning
created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
```
Indexes: `(tenant_id, created_at DESC)`, `(tenant_id, intent)`. RLS as 0185.

### `module_update_proposals` (mig 0216)
```
id                  TEXT PK
tenant_id           TEXT NOT NULL FK tenants(id)
capture_id          TEXT NOT NULL FK conversation_capture(id)
module_id           TEXT NOT NULL FK modules(id)         -- Piece B
module_template_id  TEXT NOT NULL                       -- e.g. 'ESTATE','HR'
action              TEXT NOT NULL                       -- e.g. 'create_lease_application'
payload             JSONB NOT NULL                      -- Zod-validated by module handler
confidence          NUMERIC(3,2) NOT NULL
status              TEXT NOT NULL                       -- pending_hitl|auto_applying|accepted|rejected|expired
hitl_required       BOOLEAN NOT NULL
approver_user_id    TEXT
decided_at          TIMESTAMPTZ
audit_chain_id      TEXT                                -- link to ai_audit_chain row
decision_trace_id   TEXT FK decision_traces(id)
created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
expires_at          TIMESTAMPTZ NOT NULL                -- TTL (default 72h)
```
Indexes: `(tenant_id, status, created_at DESC)`, `(tenant_id, module_id, status)`, partial `(tenant_id) WHERE status='pending_hitl'`.

### `tab_subscriptions` (mig 0217)
Static-ish registry. One row per (tenant, module_template, tab_id). Drives both realtime fan-out and the adaptive layout engine's proposal-policy.
```
tenant_id, module_template_id, tab_id, enabled, listen_intents JSONB
```

### `tab_event_log` (mig 0218)
```
id, tenant_id, user_id, persona_tier, module_id, tab_id,
event_type (view|edit|filter|export|bulk_op|record_open),
record_id, payload JSONB, ts TIMESTAMPTZ
```
Indexes: `(tenant_id, user_id, ts DESC)`, `(tenant_id, module_id, ts DESC)`. Retention: 90 days hot, then folded into semantic memory and dropped.

## Section 3 — The Capture function (deeper spec)

**Inputs:** the last user/assistant pair from `conversation_messages` (Piece F), plus `sessionContext` (persona, tier, route).
**Outputs:** one row in `conversation_capture`; zero or more in `module_update_proposals`.

**Algorithm (≤120 ms p50, Haiku-class budget):**
1. `extractEntities(text)` — regex NER (existing `entity-extractor.ts`). Confidence 0.7-0.95 by rule.
2. `resolveCanonical(entities, tenantId)` — Piece K resolver maps to `core_entity` rows. Resolver confidence multiplies the regex confidence.
3. `classifyIntent(messages, persona)` — Haiku call with prompt versioned under `packages/ai-copilot/src/prompts/v.../capture-intent.txt`. Returns intent + intent_confidence. Cached by message hash.
4. `generateActionCandidates(entities, intent, persona)` — only when intent ∈ propose_action/file_event. Reuses `plan-decomposer.ts`.
5. `score(capture)` = `min(intent_conf, resolver_conf) × persona_trust × tenant_trust`. Persona trust is highest for T1 (owner/admin) and lowest for T5 (customer).
6. Write `conversation_capture`. Below routing threshold → emit `proactive-nudge` ("I noticed you mentioned X — should I file that?"). Above → call `dispatchToTabs`.

**Distinction from Piece K:**
- Piece K = unstructured doc upload → OCR → entity extraction → routing (batch, async, doc-level audit chain).
- Piece L Hop 2 = real-time chat capture → same resolver + same routing matrix → in-line proposals.
- Both share `core_entity` resolution, the `routing_rules` matrix, and the `module_update_proposals` table. The only divergence is the source: `document_id` vs `conversation_message_id` populates exactly one of two FKs on the proposal row.

## Section 4 — Dispatcher: module routing matrix

Routing matrix is data, not code. `routing_rules` table:
```
id, tenant_id (NULL = platform default), entity_type, intent,
module_template_id, action, payload_template JSONB,
min_confidence, hitl_required, priority, created_at
```
RLS: tenant rows visible to tenant; platform defaults (`tenant_id IS NULL`) visible to all (predicate: `tenant_id IS NULL OR tenant_id = current_app_tenant_id()`). Tenant rows override platform defaults by `(entity_type, intent, priority DESC)`.

## Section 5 — Progressive Intelligence ladder

Five rungs. Tenants climb one rung at a time. Movement is **opt-in by admin** with a system-recommended threshold ("Brain has captured 200 high-confidence proposals this month with 95%+ accept rate — ready for Rung 3?"). Auto-promotion is **disabled by default** for the top two rungs.

| Rung | Phase | What signals/data | Kernel agents activated | Guardrails | How to promote |
|------|-------|-------------------|--------------------------|------------|----------------|
| 1 | Day 1 — Reactive | retrieval over migrations 0001-0185, no captures yet | think pipeline + sensors only | inviolable + policy-gate (always on) | default for new tenants |
| 2 | Week 2 — Captured | `conversation_capture` populated; `tab_event_log` accumulating | + IntentClassifier + plan-decomposer; HITL gate on every proposal | autonomy cap=0 autonomous writes; four-eye on all module mutations | flip after 50 captured-and-accepted messages |
| 3 | Month 1 — Personalised | per-tenant tone, approval routings, frequent queries; reflexion buffer (0184) hot | + Reflexion + persona-drift detector + skill-library | tenant trust factor enters scoring; HITL still required for high-stakes prefixes | admin opt-in after first weekly consolidation cycle |
| 4 | Month 3 — Predictive | semantic memory dense; cohort signal stable; forecasts grounded | + ToT + LATS + counter-model + forecasting; proactive-nudge fires | predictions APPEND not REPLACE rule-based outputs; drift-detector watches every output | admin opt-in only; requires Wave 27 eval green |
| 5 | Month 6 — Autonomous | mature tab_event_log; routing matrix tenant-tuned; standing-policy in place | + agency/executor low-stakes auto-fire; wake-loop for HQ triggers | kill-switch fail-closed; sovereign-ledger fail-closed; autonomy-policy caps daily-action count; HIGH-risk prefixes (sovereign/kill_switch/four_eye/policy_rollout) NEVER auto-fire | manual opt-in per action class; per-class cap configurable; one-month shadow run required |

Rung-3 onwards the consolidation worker rebuilds the per-tenant retrieval index every 24h. Rung 5 also requires `packages/ai-copilot/src/governance/autonomy-caps` budget set per action class.

## Section 6 — How modules consume the loop

Each module template ships three contracts, registered as a `module_template` row (Piece B):

1. **Accept-proposal schema** — Zod schemas per action.
2. **Emit-event schema** — Zod schemas per tab_event_type.
3. **In-tab copilot persona binding** — which `T-tier × persona` can use the side-rail and what tool subset.

**Worked example — ESTATE:**

`accept_proposal('create_lease_application', payload)`:
```ts
const CreateLeaseApplicationPayload = z.object({
  prospectivetenant: z.object({
    canonical_entity_id: z.string().uuid(),   // resolved Person
    full_name: z.string().min(2),
    contact_phone: z.string(),
  }),
  unit_id: z.string(),                        // resolved canonical
  desired_start_date: z.string().date(),
  monthly_rent: z.object({
    amount: z.number().positive(),
    currency_code: z.enum(['TZS','KES','UGX','NGN','USD']),
  }),
  proposed_term_months: z.number().int().min(1).max(120),
  source: z.object({
    capture_id: z.string(),
    message_id: z.string(),
    document_id: z.string().nullable(),
  }),
});
type AcceptResult = {
  application_id: string;
  status: 'draft'|'awaiting_screening'|'rejected';
  audit_chain_id: string;
};
```

Handler runs inside the agency executor's tool registry so:
- The mutation goes through `LedgerService.post()` for the deposit (rule: money path).
- A row appends to `ai_audit_chain` with `parent_hash` referencing the capture row's audit_chain_id.
- A `decision_trace` row is written.
- The realtime channel `tenant:{id}:module:ESTATE:proposals` fires.
- The side-rail emits a proactive hint to the manager: "Lease application drafted for unit 4B — review?"

Tab-event emit schema for ESTATE: `application_viewed, application_edited, application_approved, application_rejected, screening_started`.

In-tab copilot binding for ESTATE: T1-T3 personas, tool subset `{graph.query, forecast.arrears, kb.search, docs.search}`.

## Section 7 — Integration with Piece F (threads) and Piece G (artifacts)

A `conversation_thread` (Piece F) can produce TWO outputs from a single user message:
1. **Visual artifact** via Piece G — a `ui_artifacts` row referenced inline in the assistant message (re-uses `packages/chat-ui/src/generative-ui/AdaptiveRenderer.tsx`).
2. **Tab-routed capture** via Piece L — `conversation_capture` row + `module_update_proposals` rows.

**Worked contract example:** *"Find me all leases ending in Tabora in Q4."*

```
conversation_messages.id = m1 (user msg)
conversation_messages.id = m2 (assistant msg, references artifact a1)

ui_artifacts:
  id=a1, type='data_table',
  payload={columns:[...], rows:[...]},
  message_id=m2

conversation_capture:
  id=c1, message_id=m2, intent='request_info',
  entities=[{type:'jurisdiction',value:'Tabora'},{type:'period',value:'2026Q4'}]

module_update_proposals:
  id=p1, capture_id=c1, module_template_id='ESTATE',
  action='bulk_mark_for_renewal_prep',
  payload={lease_ids:[...]},
  status='pending_hitl' (bulk ops always HITL),
  confidence=0.86
```

The artifact answers; the proposal acts. The user sees both in one thread but only one (the proposal) reshapes a tab. The artifact's `payload.rows` and the proposal's `payload.lease_ids` are joined on `core_entity_id` to guarantee both refer to the same canonical leases.

## Section 8 — Audit + safety

Every hop is OTel-traced AND audit-chained. The forensic chain for any tab change is:

```
ai_audit_chain (kernel decision)
  ↓ parent_hash
conversation_capture
  ↓ FK
module_update_proposals
  ↓ FK
decision_traces  (executor decision)
  ↓ correlation
tab_event_log    (user reaction)
```

Replay: given any brief that proposed an action, the auditor calls `services/api-gateway/src/routes/audit-trail.router.ts` with the action's `audit_chain_id`. Server walks back through hash-chain to the capture, to the message, to the kernel `think()` inputs (sensors, memory hits, prompt version, model hash) — fully reconstructible.

OTel span hierarchy: `brain.think` → `brain.capture` → `dispatch.route` → `module.accept_proposal` → `ledger.post` (when money). Every span tagged with `tenant_id`, `persona_tier`, `module_template_id`, `capture_id`. PII scrubbed by the existing `kernel/cot-reservoir.ts` PII-extended scrubber before span export.

Inviolable on the loop: capture and dispatch run AFTER `checkInviolable()` and `runPolicyGate()` in the existing pipeline. A refused/softened decision NEVER produces a capture row (skip capture when `BrainDecision.kind !== 'answer'`).

## Section 10 — Risks + mitigations

| # | Risk | Mitigation |
|---|------|------------|
| 1 | **Cross-module / cross-tenant data leakage** through dispatch (a capture from tenant A creates a proposal on tenant B's module) | FORCE RLS on every new table; dispatcher constructs queries through tenant-scoped Drizzle client only; integration test `dispatcher-tenant-isolation.test.ts` mandatory before each wave merge; auditor scan on every CI run. |
| 2 | **Runaway autonomous writes** at Rung 5 (executor accepts proposals faster than humans can audit) | Three layers: per-action-class daily caps in `autonomy-caps`; kill-switch fail-closed (already exists, never `try/catch`-ignore); sovereign-ledger fail-closed for HIGH-risk prefixes (already in `executor.ts`); plus a new global `proposals_accepted_today` counter with hard ceiling. |
| 3 | **Capture hallucination** (intent classifier invents entities not in the message) | Resolver MUST return a canonical `core_entity_id`; if not resolvable, entity is dropped, not stored. Capture confidence is `min(...)` not `avg(...)` — one weak link kills it. Eval suite samples 5% of captures for offline judge re-scoring. |
| 4 | **Tab desync** (realtime channel drops, UI shows stale data) | React Query stale-time + window-focus refetch on all `module_update_proposals` keys; heartbeat ping on Supabase channel; offline banner via existing `packages/chat-ui/src/components/DegradedBanner.tsx`. |
| 5 | **Audit chain breakage** (one hop fails to write, parent_hash dangles) | Each hop writes through the existing `ai_audit_chain` append, which is transactional. Dispatcher refuses to create a proposal if the capture's audit row is missing. Nightly chain-integrity check job (extend `services/consolidation-worker/src/stages/08-publish.ts`). |
| 6 | **Latency** (capture + dispatch adds tens of ms to every reply) | Capture runs AFTER the kernel streams its answer to the user — async, off the critical path. Dispatcher and executor are fire-and-forget from the user's perspective; proposals appear via realtime when ready. p50 capture target: ≤120ms. p99: ≤500ms. |

## 5 architectural decisions to lock now

1. **Capture is post-pipeline, not in-pipeline.** It runs AFTER `kernel.think()` finalises, on the (user_msg, assistant_msg) pair, so it never adds latency to the user's reply. Refused / softened decisions skip capture entirely.
2. **Routing matrix is data in `routing_rules`, not code.** Platform defaults are NULL-tenant rows; tenants override per (entity_type, intent, priority). Ship matrix changes as data migrations, not code deploys.
3. **Every module mutation goes through `accept_proposal(proposal_id)`, no direct writes.** This makes the executor the single point that enforces four-eye, sovereign-ledger, and inviolable for the entire loop.
4. **Progressive intelligence is admin-opt-in for Rungs 4-5, never auto-promoted.** Rungs 1-3 promote on simple thresholds; Rungs 4-5 require explicit admin click + a one-month shadow-mode run.
5. **Capture and Piece K share ONE resolver and ONE routing matrix.** Doc upload and chat capture produce identical `module_update_proposals` rows — only the source FK (`document_id` vs `message_id`) differs.
